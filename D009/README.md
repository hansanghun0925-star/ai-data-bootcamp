# 빵셀러 데이터 정제·집계 보고서

## 1. 데이터 개요
- 출처: (가상) 빵셀러 베이커리 체인 운영 로그
- 기간: 2025-04-01 ~ 2025-06-30 (3개월)
- 원본 행 수 / 정제 후 행 수: (2164)행 → (1784)행

## 2. 발견한 품질 문제
- 결측: revenue 컬럼에 60건. 결측비율 2.77%
- 중복: 완전 중복 행 321 건
- 이상치(IQR 기준): quantity 14.56%, revenue 17.3%  (결측이 있어 정제 후 재확인 필요)
- 표기 혼재: store_id에 앞뒤 공백(" S01 "), item에 케익/케이크 혼용

## 3. 처리 결정과 근거 (5줄로)
1. 중복 행 (321)건 → 제거               : 매출 합계 분석이 목표이기 때문
2. store_id 공백 → 앞뒤 공백 제거       : ' S01 '과 'S01'은 같은 매장
3. item '케익' → '케이크'               : 올바른 표준어 표기인 '케이크'로 바꿈 표기 혼재는 집계를 쪼개기 때문
4. revenue 음수 → 별도 보관 (1)건       : 환불로 판단 
5. revenue 결측 (60)건 → 제거           :추후 평균을 내거나 할때 행 개수와 값이 맞지 않는 경우가 생길 수 있으므로 제거. (하지만 quantity 열과 unit_price 열이 정상적으로 존재하기에, 곱하기해서 제거하지 않는 방향으로 분석을 진행할 수 도 있다.)

## 4. 주요 KPI 결과 (2줄)
- (월·매장 관점) 가장 매출이 큰 매장과 그 월:    S06 매장 /  2025년 4월
- (상품 관점) 매출 1위 상품: total_revenue 기준, 쿠키가 26,690,000원으로 1위

## 5. 한계와 후속 작업
- 단순 곱으로만 revenue를 산출하기에는, 실제 현장에서의 복합적인 상황을 고려못할 가능성이 존재함.  
- revenue 열을 quantity열과 unit_price열의 곱으로 계산했는데, 이상치 비율이 다르게 나오는 이유 알아보고 싶다. - 곱으로 정의되었기 때문에, quantity의 50이 비싼 unit_price를 만나면 이상치를 증폭시켜 revenue 의 이상치 비율이 더 높음. 
- 메뉴판매량 추이를 매장의 위치와 시기에 따라 한번 관찰하고 싶다. 


| 단계 | 액션 | 근거 | 결과 |
|---|---|---|---|
| 1. 중복 제거 | 완전 중복 행 drop | 동일 행 반복은 입력 오류로 간주 | 중복 4건 제거 |
| 2. store_id 공백 정제 | str.strip() | ' S01 '과 'S01'은 같은 매장 | 공백 제거로 매장 통일 |
| 3. item 표기 통일 | '케익'→'케이크' replace | 표기 혼재는 집계를 쪼갬 | item 종류 통일 |
| 4. 날짜 파싱 | date_str→datetime (format='mixed') | 월별 집계에 datetime 필요 | 혼합 포맷 파싱 |
| 5. 환불·결측 처리 | 음수 revenue 분리 + 결측 행 제거 | 매출 분석 목적, 환불은 별도 보관 | 환불 1건 분리 |
